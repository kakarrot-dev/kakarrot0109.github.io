# 功能规格：Claude Code 与 Codex 项目规范博客

## 背景

同一个项目同时使用 Claude Code 和 Codex 时，`CLAUDE.md`、`AGENTS.md` 以及两套工具的专属配置容易形成重复维护。现有《Claude-Code vs Codex 项目规范对照表》覆盖面较全，但更接近资料手册，缺少个人实践主线，也包含需要根据当前官方资料重新核验的版本敏感信息。

## 目标

- 将原始对照表整理为网站 Writing 栏目的 `No. 21` 文章。
- 面向已经同时使用 Claude Code 和 Codex、正被重复配置困扰的读者。
- 以个人实践为主线，回答如何减少重复维护，同时保留两套工具必要的差异。
- 在叙事之外恢复可复制的配置示例和精简对照手册，让文章具备日后查阅价值。
- 提供一套读者可以直接参考的最小项目目录、加载顺序与维护原则。

## 非目标

- 不逐项复刻原文全部配置说明。
- 不写成 Claude Code 或 Codex 的完整官方手册。
- 不声称两套工具的配置文件、Skills、Hooks 或子 Agent 可以完全通用。
- 不编造作者未提供的具体事故、团队规模、使用时长或量化收益。
- 不修改网站布局、Content Collections schema 或其他文章。

## 用户场景

读者在同一仓库中同时使用 Claude Code 和 Codex，发现项目背景、构建命令和开发约束需要维护两份，希望找到既能减少重复、又不混淆平台专属能力的组织方式。

## 需求边界

### 包含

- 个人叙事开场：同时维护两套项目规范带来的重复与不一致。
- 核心观点：项目事实保持单一来源，平台入口只承担适配职责。
- 区分可共享内容与必须分开的平台配置。
- 两张精简表：一张 Claude Code / Codex 核心文件对照表，一张三层内容职责与推荐位置表。
- 三个核心代码块：`CLAUDE.md` 薄入口、两端简化加载顺序、推荐目录结构。
- 一份推荐的最小目录结构和维护原则。
- 对加载规则、Skills、Hooks、子 Agent 等版本敏感事实进行当前官方资料核验。
- 延续网站文章的序号、口语化叙事、emoji 编号主体和文末标签格式。

### 不包含

- 原文全部目录树和字段级配置示例。
- 未经核验的跨工具兼容性承诺。
- 新增插图、交互组件或下载附件。
- 自动同步 `CLAUDE.md` 与 `AGENTS.md` 的脚本。

## 交互流程

1. 读者从同时维护两套规范的真实困扰进入文章。
2. 先理解重复维护的根因不是“文件太多”，而是项目事实没有明确归属。
3. 通过核心文件对照表确认两端机制并非一一对应。
4. 通过三层职责表识别哪些内容可以共享、哪些必须保留平台差异。
5. 复制薄入口示例，并参考加载顺序和最小目录调整自己的仓库。
6. 用文章末尾的维护原则检查后续规则应该放在哪里。

## 数据与状态

- 新增一篇 `src/content/writing/` 下的 Markdown 文章。
- frontmatter 使用现有 schema：
  - 标题：`No. 21: 一套项目，两套 Agent——我如何同时维护 Claude Code 和 Codex`
  - 发布日期：`2026-07-20`
  - 标签：`Claude Code`、`Codex`、`Coding Agent`、`项目规范`，并与文末标签一致。
- 不新增运行时状态、缓存或本地存储。

## 权限与安全

- 不在示例中出现真实 Token、Cookie、私钥或本机敏感路径。
- 配置示例只展示目录职责，不提供可能被误用的权限放开配置。
- 官方事实优先引用官方文档；无法确认的能力明确标注版本差异或从正文删除。

## 异常情况

- 如果原文说法与当前官方文档冲突，以当前官方文档为准，并在文章中避免保留过时结论。
- 如果某项能力只能在特定版本或实验功能中使用，不把它写成稳定通用能力。
- 如果缺少作者真实经历，只写已确认的“同一项目同时使用两套工具并产生重复维护”，不扩写虚构细节。

## 验收标准

- 新文章序号为 `No. 21`，frontmatter 通过 Content Collections 校验。
- 文章采用个人实践主线，而不是把原始对照表换一种排版。
- 正文包含：问题起点、核心判断、共享与分离边界、两张精简表、三个核心代码块、最小目录、维护原则。
- Markdown 表格总数固定为 2 张，fenced code block 总数固定为 3 个；保留查阅价值，但不逐段复刻原始手册。
- 只保留经当前官方资料核验的版本敏感事实，并附必要来源链接。
- 文末标签与 frontmatter 一致。
- 内容修订后重新运行 `npm run check`、`npm run lint`、`npm run build`，并更新 No. 21 详情页构建产物；首次 `check` 出现的同 slug duplicate id 瞬时提示需结合源文件唯一性和后续检查结果核对。
- 构建产生的首页与 Vibe Coding 无关排序变化按用户确认的方案 A 恢复，只保留 No. 21 详情页的相关构建更新。
- 检查 README.md 与 `tasks/lessons.md` 是否需要同步，不手工编辑 `docs/` 构建产物；相关构建产物和任务记录已提交，当前分支已正常推送。
