# Kakarrot Astro 个人网站设计

## 目标

将现有 Hexo 博客作为已备份的历史项目处理，不迁移旧文章和旧链接。从零构建一个面向产品经理、创业者和教育业务负责人的个人网站。

网站的核心定位是：

> 我是 Kakarrot，一名 AI 教育产品经理。我从第一性原理拆解复杂教育业务，用 AI 原生方式把它做成可落地的产品系统。

网站不是个人履历或博客归档，而是一个判断入口：访客应在 30 秒内确认 Kakarrot 懂教育业务、懂 AI 产品架构，并能把复杂问题讲清楚、做出来。

首版同时承担两项任务：

1. 通过首页、项目案例和关于页建立个人品牌。
2. 通过独立文章系统承载长期写作。

## 范围

首版包含：首页、项目列表、项目详情、文章列表、文章详情和关于页。

首版不包含：旧 Hexo 内容迁移、旧链接兼容、CMS、评论、账号系统、站内搜索、在线工具和实验室。实验室作为二期方向保留，但首版导航和首页不放置空入口。

## 技术架构

采用纯 Astro 静态站：

- Astro 负责路由、页面模板、静态生成和 SEO。
- Astro Content Collections 管理文章与项目案例。
- 内容使用 Markdown，不引入 Headless CMS。
- 不引入 React、Vue 等客户端框架。
- 基础交互使用 CSS；只有无法用 CSS 完成的必要交互才使用少量原生 JavaScript。
- 所有正式页面在构建阶段生成，访问时不依赖服务端运行时。

选择该方案的原因是：网站的核心价值来自内容和表达，不来自复杂交互。纯静态架构可将运行成本、维护成本和故障面降到最低。

## 信息架构

| 路径 | 用途 |
|---|---|
| `/` | 个人定位、当前关注、方法、精选项目、精选文章、可信证据、简介 |
| `/projects/` | 项目案例列表 |
| `/projects/[slug]/` | 项目案例详情 |
| `/writing/` | 文章列表与主题浏览 |
| `/writing/[slug]/` | 文章详情 |
| `/about/` | 经历、原则、网站背景和联系方式 |

首页按照以下顺序叙事：

1. 我是谁：AI 教育产品经理，解决复杂教育业务问题。
2. 我现在关注什么：高校 AI 场景、教育集团平台、Agent 工作流和 AI 原生产品架构。
3. 我怎么做：定位问题本质、拆解业务系统、用 AI 原生方式重构。
4. 我做过什么：用项目案例和可公开产物证明实践能力。
5. 我如何思考：用精选文章展示判断过程。
6. 为什么可信：展示经历、长期实践、产品文档、架构图、Demo 或公开代码等证据。

首页不以“我写过什么”为主线，而以“我现在能解决什么问题”为主线。文章、项目和个人介绍都服务于该判断。

## 视觉设计

整体采用“专业的 AI 产品工作台”概念：首页和项目页使用克制的新粗野主义与结构化工作台语言，文章页使用编辑杂志风。借鉴参考站的人格叙事，不复制设计师作品集的手账皮肤和高密度动效。

设计规则：

- 主色为黑、灰和低饱和暖底色，酸性绿只用于少量重点标记和交互反馈。
- 中文为主体；英文只用于导航、编号和辅助信息。
- 首页使用高对比、大字号和明确网格，但不复制参考站的超大姓名设计。
- 首屏采用第一人称表达，避免咨询公司或设计机构语气。
- 图形语言来自 PRD、架构图节点、Prompt 卡片、终端窗口和组织流程图；每个元素必须承载真实信息，不作为装饰堆叠。
- 项目区域优先展示真实证据；不能公开的内容使用流程图、系统图和抽象示意表达。
- 文章正文宽度控制在 760–820px，提供目录、发布时间、更新时间、标签和相关文章，弱化边框和强调色。
- 不使用圆角卡片和装饰性阴影。
- 动效限制为 CSS 悬停、必要的文字移动与轻量淡入，并尊重 `prefers-reduced-motion`。
- 移动端取消装饰性横向排版，优先保证内容顺序和可读性。

视觉草图用于确认方向，不作为实现代码直接复用。

## 页面与组件边界

- `BaseLayout`：统一元数据、全局导航和页脚。
- `HomeHero`：个人定位、首屏说明和身份信息。
- `FocusSection`：展示当前关注的业务和技术问题。
- `MethodSection`：展示三层方法，不承担案例详情。
- `ProjectCard`：展示项目摘要、主题和入口。
- `ArticleCard`：展示文章标题、摘要、主题和日期。
- `EvidenceList`：展示可公开的文档、架构图、Demo、代码或行业实践证据。
- `ProseLayout`：约束文章和案例正文排版。
- `TableOfContents`：根据正文标题静态生成文章目录。
- `SeoHead`：生成 Canonical、Open Graph 和基础元数据。

组件只接收展示所需数据。内容查询和排序由页面在构建阶段完成，避免组件同时承担数据读取和渲染职责。

## 内容模型

文章至少包含：`title`、`description`、`publishedAt`、`updatedAt`、`tags`、`draft`、`featured`。

项目至少包含：`title`、`description`、`publishedAt`、`topics`、`featured`、`draft`、`disclosureLevel`、`evidence`。

Content Collections 使用 Schema 校验字段。字段缺失或格式错误时构建失败，不生成不完整页面。草稿在生产构建中排除。

## 首个项目案例：悟空

悟空采用公开级别 C：只公开思考过程，不公开真实界面、代码、技术细节和业务数据。

案例结构：

1. 业务问题：为什么需要悟空。
2. 错误起点：为什么聊天机器人不是答案。
3. 第一性原理：用户实际需要完成的任务。
4. 业务拆解：角色、输入、决策、执行和反馈。
5. AI 原生方案：智能体如何重组流程。
6. 实践反思：判断、取舍和仍需解决的问题。
7. 公开边界：明确说明敏感实现和业务数据未披露。

视觉材料只使用流程图、系统图、决策树和抽象示意，不虚构结果数据。

## 可信证据规则

首页和项目页不使用无法验证的自我评价。每项能力优先由以下证据支撑：

1. 可公开的项目案例和产品原型。
2. 脱敏后的业务流程图、产品架构图和决策记录。
3. 公开文章、演示文稿、代码仓库或工具成果。
4. 明确标注背景和边界的个人实践反思。

证据不足时减少结论强度，不使用虚构客户、数据、奖项或结果。

## 二期方向

首版稳定后再评估 `/lab/`。实验室可承载 AI 原型、Prompt 模板、UI 组件和工具链测试。实验内容可独立部署，但主站只保存索引和说明，避免把动态实验的技术复杂度引入 Astro 主站。

## SEO 与发布

- 每个页面拥有独立标题和描述。
- 生成 Canonical、Open Graph、Sitemap 和 RSS。
- 静态图片保存在仓库中，构建时生成适合不同尺寸的资源。
- 仓库只使用 `main` 作为核心源码分支；Astro 源码、Markdown 内容和项目文档均保存在该分支。
- GitHub Actions 负责安装依赖、校验、构建并发布到 GitHub Pages。
- 构建后的 `dist/` 作为 GitHub Pages Artifact 上传，不提交到 Git，也不创建 `gh-pages`、`source` 或 `master` 发布分支。
- 自定义域名继续使用 `kakarrot.com`，发布产物包含 `CNAME`。
- 部署后必须验证域名解析、HTTPS、静态资源路径和 404 页面。

## 错误处理

- 内容字段错误：构建失败并显示具体文件和字段。
- 内部链接失效：链接检查失败，阻止发布。
- 图片缺失：构建或资源检查失败，阻止发布。
- 页面不存在：返回自定义静态 404 页面。
- GitHub Pages 发布失败：保留上一版线上产物，不覆盖现有网站。

## 验证与成功标准

- 未接触过 Kakarrot 的目标读者能在 30 秒内说出其身份、业务领域、方法和一项可信证据。
- 首页首屏明确表达“AI 教育产品经理”，并说明能解决的业务问题。
- 所有正式页面均可静态生成，默认不加载客户端框架。
- 类型检查和生产构建通过。
- 内部链接和静态资源检查通过。
- 桌面端与移动端无横向溢出，导航和正文可正常阅读。
- 当前版本 Chrome、Safari、Firefox 可正常访问。
- Lighthouse 的性能、SEO、无障碍分数均不低于 90。
- 悟空案例不包含真实界面、代码、敏感技术实现或业务数据。
- `kakarrot.com` 自定义域名和 HTTPS 正常。
