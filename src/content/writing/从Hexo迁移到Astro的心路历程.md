## 一场分析引发的迁移

事情要从头说起。

这个博客从 2016 年开始跑 Hexo，前后换过三四套主题，积累了 17 篇文章。习惯了，也没想过挪。

直到有一天，在x上看到一些灵感，把博客网站打造成个人作品站点，加上vibe coding的经历 开始折腾吧。

## 问题在哪

不是 Hexo 不够好。是**我的站已经不是一个"个人网站"，而是一个"文章收纳盒"**。

对比下来，差距在三个层面：

**信息密度。** 人家的首屏像一张创作者桌面：纸条、咖啡、铅笔、终端窗口、文件夹、项目卡片。访客一眼就知道"这个人会做什么"。我的首屏：名字 + 导航。访客只能去猜。

**叙事结构。** 人家是 `About / Work / Playground`，下面再拆项目、AI 设计、社区影响、摄影。每个板块都在回答"我能解决什么"。我是 `归档 / 分类 / 标签 / 关于`。新读者进来，不知道该看哪篇。

**可信证据。** 人家有项目案例、Awwwards 提名、Vibe Coding 实验入口。我有 17 篇文字，没有一条说"这人做过的产品是什么样的"。

问题不在 Hexo，问题在于：**我把"写博客"当成了个人网站的全部功能。**

## 为什么是 Astro

目标很明确：从"博客归档"变成"个人品牌入口"。需要支持文章、项目案例、交互实验、个人定位，且长期维护成本尽可能低。

看了一圈：

- **继续 Hexo** —— 要自定义页面结构，需要改模板和插件，改造深度跟迁移差不多
- **Hugo** —— 快，但模板语法我完全不想学
- **Next.js** —— 适合高交互站点，对我这种以内容为主的站来说太沉
- **Astro** —— 静态优先、原生 Content Collections、文件路由、零 JS 默认输出

Astro 的 Content Collections 是最打动我的点：用 Schema 定义 frontmatter，类型安全，构建时校验。写文章就是写 Markdown，没有额外的数据层。

## 迁移做了什么

- 从 Hexo `source/_posts/` 搬到 Astro `src/content/writing/`，统一 frontmatter schema
- 重新设计布局：shell 容器、固定宽度、全屏色块穿透
- 配色改用暖白底色 + 酸绿色强调色（#39ff14），取代之前灰白老主题
- 做了项目页、About 页、RSS、404 —— 之前 Hexo 时代依赖于主题自带，现在自己控制
- 滚动渐入动效、文章 typography 整套重写
- 部署从 Hexo `master` 分支部署改为 GitHub Actions + `main` 分支

最花时间的不是迁移文章，是**重新想清楚这个站应该长什么样**。

## 迁移前后对比

| 维度 | 之前（Hexo） | 之后（Astro） |
|------|------------|-------------|
| 定位 | 博客归档 | 个人品牌入口 |
| 内容结构 | 归档 / 分类 / 标签 | 文章 / 项目 / 实验室 / 关于 |
| 文章管理 | 纯 Markdown 文件 | Markdown + Schema 校验 |
| 路由 | `_config.yml` + 插件 | 文件系统路由 |
| RSS | 需要 `hexo-generator-feed` | 一个 `.ts` 文件 |
| 自定义能力 | 改主题模板 | 写 `.astro` 组件 |
| 部署 | `master` 分支静态文件 | GitHub Actions + workflow |
| 构建产物 | 纯静态 HTML | 纯静态 HTML（无 JS runtime） |

## 几点感受

**迁移本身不难。想清楚要什么最难。**

如果我只是把 17 篇文章从 Hexo 复制到 Astro，把界面重新画一遍——那什么都不会改变。真正的迁移发生在脑子里：从一个"我写过什么"的归档思维，变成一个"我现在能解决什么问题"的品牌思维。

**静态站不需要框架化。** Astro 的输出还是纯 HTML + CSS，跟 Hexo 一样干净。它没有让这个站"变现代"，它只是让"写网站的体验"现代化了。

**一个好的参考对象不是用来抄的。** yanliudesign.live 是全视觉向的设计师站。我是 AI 教育行业产品经理，站点的第一原则应该是：让别人快速判断——你是否懂 AI、懂教育业务、懂产品架构、能把复杂系统讲清楚并落地。所以我借鉴的是它的叙事方式，不是它的视觉皮肤。

## 现在

这个站不会停在这次迁移。接下来会陆续补上项目案例、AI 教育产品实验室。但重点回到内容创作本身——站是容器，文章和产品才是内容。

**一个个人站应该像一份持续更新的说明书：告诉访客你是谁、做什么、能解决什么问题。而过去十年，我一直在写目录，没有写过说明书。**
